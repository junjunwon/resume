서비스 설명

PHP 기반 엑셀 발주 배치를 Spring Batch로 전환한 프로젝트입니다. PHP 시스템에서 15,000개 데이터 처리 시 12분 소요 또는 OOM 발생 문제가 있었으며, Spring Batch 전환 초기 동일 문제가 발생했습니다. 개별 조회 및 처리를 Bulk 처리로 전환하고 배치 단위 메모리 해제를 구현하여 OOM 문제를 해결한 프로젝트입니다.

책임

  • Spring Batch 아키텍처 설계 및 구현 (Job, Step, Chunk 기반 처리 흐름 설계)
  • 성능 최적화 (Bulk 처리 전환, N+1 쿼리 해결, 메모리 효율 개선)
  • 안정성 강화 (중복 처리 방지, 트랜잭션 관리, Fault Tolerance 구현)
  • 메모리 프로파일링 기반 최적화 (IntelliJ Profiler 활용)

주요 기여

성능 개선

  • 처리 시간 93.75% 감소: 12분 → 45초 (16배 속도 향상) — Spring Batch 전환 후에도 동일 문제 발생 → 개별 조회 및 처리 부분을 전부 Bulk 처리로 변경, Row 단위 처리 → Bulk 처리 전환, N+1 쿼리 해결: 30,000번 → 1번 (99.997% 감소)
  • PHP 및 Spring Batch 초기 버전에서 발생하던 OOM 문제 해결 — BatchInsertExecutor 유틸리티 클래스 개발로 배치 단위 메모리 해제 구현, Peak 메모리 37.5% 감소 (400MB+ → 250MB)

아키텍처 개선

  • 5개 독립 프로세스 → 1개 통합 Job: 프로세스 관리 단순화 및 실행 주기 통일
  • 4단계 Step 기반 모듈화: 회사 목록 조회 → 상태 업데이트 → 엑셀 읽기 → 주문 생성
  • 파일 단위 독립 처리: 부분 실패 격리로 영향 범위 최소화
  • 중복 처리 방지: Step 2에서 원자적 상태 변경 및 Commit으로 배타 제어

7. 클러스터간 PK 중복없이 회사를 이관하기 위한 다우오피스 전체 클러스터 채번 전략 재수립

회사: 다우기술대상 서비스: 다우오피스 SaaS 서비스

책임

  • 클러스터 단위로 PK Generator를 적용하기 위한 채번 전략 재수립 및 구현
  • AS-IS 다우오피스 PK 전략 변경 후 Cloud/VPC 등 환경별 테스트
  • PK 마이그레이션 이후 DB에 String 타입으로 저장된 마이그레이션이 불가능한 PK에 내부링크 적용 — 내부링크: 마이그레이션 이전 PK를 담은 Model을 Client에게 전달할 때 마이그레이션 PK로 변경

주요 기여

  • 장기간 사용할 수 있는 Snowflake 기반 분산 PK의 비트단위 정책 수립 (With DBA) — timestamp 41비트, sequence 12비트 → 69.73년동안 workerId당 1ms당 4096까지 중복없이 사용가능하도록 설계. workerId는 클러스터 +10개 & +수백개의 대형 VPC 고객사를 고려하여 10비트로 확정
  • 클러스터 단위로 PK Generator 전략 구현하여 유연한 채번전략을 활용할 수 있도록 설계 — 채팅용 Sequence과 기본 PK 전략인 Table Generator 커스텀 및 신규 Snowflake 구현체 구현. 각 구현체는 IdentifierGenerator, Configurable 구현한 CustomIdentifier 인터페이스를 상속받아 PK 전략별 configure()와 generate() 구현. @GenericGenerator + CustomIdentifier를 상속받은 CustomGenerator 클래스를 구현하여 클러스터 구동 시점에 프로퍼티에서 조회한 PK 전략에 맞는 configure 구현체 호출 및 PK 설정정보 세팅

상세 설명

9개 클러스터(클러스터별 8억개의 PK 보유)와 12,000개 고객사에서, 클러스터별로 회사 PK가 중복 증가하여 클러스터 간 회사 이동이 불가능하고, 이로 인해 부하 집중 및 리소스 재활용 불가 문제가 확인됨. 기존 클러스터는 기존 PK 전략을 유지하고, 신규 클러스터로의 이관 시 마이그레이션 PK 규칙을 적용하며, 신규 클러스터에는 Snowflake 채번 방식을 병렬 적용하기 위해 클러스터별 PK Generator를 개발 및 적용하는 것이 목표. 테스트서버에서 다우메신저와 그룹웨어 등 전체 서비스에 영향을 미치기 때문에 모든 기능에 대한 전수 테스트 진행 완료. 채번 전략 설계 및 테스트서버 구현 & 테스트는 모두 진행하였으나 실제 당시 운영 배포는 HOLD — 사유: 현 다우오피스에 적용하지 않고, 25년 하반기 개발완료될 차세대 다우오피스에 해당 채번 전략을 적용하는 것으로 계획 변경 (개발시점: 24년 5월). 25년도 상반기에 차세대 다우오피스 개발팀에 인수인계 진행 (설계문서, 당시 PR과 리뷰 기록, WBS 등).

협업: DBA, 메일 엔지니어, QA 엔지니어, 백엔드 개발자 다수기술 스택: SpringBoot, Hibernate, JPA, Java, Bitbucket(Git), Jira
