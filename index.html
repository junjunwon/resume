서비스 설명

PHP 기반 엑셀 발주 배치를 Spring Batch로 전환한 프로젝트입니다. PHP 시스템에서 15,000개 데이터 처리 시 12분 소요 또는 OOM 발생 문제가 있었으며, Spring Batch 전환 초기 동일 문제가 발생했습니다. 개별 조회 및 처리를 Bulk 처리로 전환하고 배치 단위 메모리 해제를 구현하여 OOM 문제를 해결한 프로젝트입니다.

책임

  • Spring Batch 아키텍처 설계 및 구현 (Job, Step, Chunk 기반 처리 흐름 설계)
  • 성능 최적화 (Bulk 처리 전환, N+1 쿼리 해결, 메모리 효율 개선)
  • 안정성 강화 (중복 처리 방지, 트랜잭션 관리, Fault Tolerance 구현)
  • 메모리 프로파일링 기반 최적화 (IntelliJ Profiler 활용)

주요 기여

성능 개선

  • 처리 시간 93.75% 감소: 12분 → 45초 (16배 속도 향상). 개별 조회·처리를 전부 Bulk 처리로 전환, N+1 쿼리 해결: 30,000번 → 1번 (99.997% 감소)
  • PHP 및 Spring Batch 초기 버전에서 발생하던 OOM 해결 — BatchInsertExecutor로 배치 단위 메모리 해제 구현, Peak 메모리 37.5% 감소 (400MB+ → 250MB)

아키텍처 개선

  • 5개 독립 프로세스 → 1개 통합 Job: 프로세스 관리 단순화 및 실행 주기 통일
  • 4단계 Step 기반 모듈화: 회사 목록 조회 → 상태 업데이트 → 엑셀 읽기 → 주문 생성
  • 파일 단위 독립 처리: 부분 실패 격리로 영향 범위 최소화
  • 중복 처리 방지: Step 2에서 원자적 상태 변경 및 Commit으로 배타 제어

7. 클러스터간 PK 중복없이 회사를 이관하기 위한 다우오피스 전체 클러스터 채번 전략 재수립

회사: 다우기술대상 서비스: 다우오피스 SaaS 서비스

책임

  • 클러스터 단위로 PK Generator를 적용하기 위한 채번 전략 재수립 및 구현
  • AS-IS 다우오피스 PK 전략 변경 후 Cloud/VPC 등 환경별 테스트
  • PK 마이그레이션 이후 DB에 String 타입으로 저장된 마이그레이션이 불가능한 PK에 내부링크 적용 — 내부링크: 마이그레이션 이전 PK를 담은 Model을 Client에게 전달할 때 마이그레이션 PK로 변경

주요 기여

  • Snowflake 기반 분산 PK 비트 단위 정책 수립 (With DBA) — timestamp 41비트, sequence 12비트로 약 69년간 workerId당 1ms당 4,096개까지 중복 없이 사용 가능. workerId 10비트로 클러스터·대형 VPC 고객사 확장 고려
  • 클러스터 단위 PK Generator 전략 구현 — 채팅용 Sequence, Table Generator 커스텀 및 Snowflake 구현체 구현. CustomIdentifier 인터페이스·@GenericGenerator 기반으로 클러스터 구동 시 프로퍼티에 따른 PK 전략 적용

상세 설명

배경 — 9개 클러스터(클러스터당 약 8억 PK), 12,000개 고객사. 클러스터별 회사 PK가 중복 증가해 클러스터 간 회사 이동이 불가했고, 부하 집중·리소스 재활용 불가 문제가 있었음.

목표 — 기존 클러스터는 기존 PK 전략 유지, 신규 클러스터 이관 시 마이그레이션 PK 규칙 적용, 신규 클러스터에는 Snowflake 채번을 쓰기 위해 클러스터별 PK Generator 개발·적용.

수행 — 채번 전략 설계, 테스트서버 구현 및 다우메신저·그룹웨어 등 전체 서비스 전수 테스트 완료. 당시 운영 배포는 HOLD(현 다우오피스 미적용, 25년 하반기 차세대 다우오피스 적용 예정으로 계획 변경, 개발 시점 24년 5월). 25년 상반기 차세대 다우오피스 개발팀에 설계문서·PR·리뷰 기록·WBS 등 인수인계 완료.

협업: DBA, 메일 엔지니어, QA 엔지니어, 백엔드 개발자 다수기술 스택: SpringBoot, Hibernate, JPA, Java, Bitbucket(Git), Jira
